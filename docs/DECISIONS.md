# 決定事項と検討経緯 — レシピヘルパー

これまでのチャットでの検討内容の記録。Claude Codeはこの経緯を前提として作業すること。

## 決定済み

| 項目 | 決定 | 理由・経緯 |
|---|---|---|
| アプリ名 | レシピヘルパー(recipe_helper) | |
| アプリ形式 | PWA(単体HTML) | 過去作(ichiban-uranai, tabi-saifu等)と同方式 |
| レシピデータ元 | アプリ内に同梱する自作データ(外部APIなし) | 完全無料・オフライン方針を維持するため |
| レシピ規模 | 数百件を目指すが目安。件数よりマッチング精度を優先 | ユーザーが精度優先を明示 |
| 食材カテゴリ | 5つ(野菜/肉・魚/乳製品[卵含む]/加工食品/主食・粉類) | 卵は当初独立検討したが、最終的に乳製品に統合して確定 |
| マッチング方式 | 選択食材と `main_ingredients` の一致数でスコアリング、上位3件以上表示 | |
| 開発の場 | Claude Code | レシピ数が多く、チャットでの企画確定→Code側で本開発という流れ |

## マッチング精度に関する方針(重要)

- 件数を追うより、食材マスタとタグの表記統一を優先する
- 食材マスタ(選択肢の正式名称リスト)を先に確定し、そのマスタの語彙だけでレシピにタグ付けする
- ジャンル・調理法に幅を持たせ、同じ食材セットでも複数系統のレシピが出るようにする

## 開発開始後の決定事項(2026-07-04)

| 項目 | 決定 | 理由・経緯 |
|---|---|---|
| 食材マスタ | 90品で確定(野菜32/肉・魚20/乳製品7/加工食品20/主食・粉類11)。`data/ingredients.js` に収録 | |
| にんにく・しょうが | マスタ対象外(調味料扱い) | ほぼ全レシピに登場しマッチングのノイズになるため |
| 米とごはん | 別項目として両方収録 | 「米」=炊く前(炊飯を伴う料理)、「ごはん」=炊いたごはんをすぐ使う料理(チャーハン・丼・雑炊など) |
| チーズ表記 | スライス・ピザ用・とろけるは「チーズ」に統合。「粉チーズ」のみ別項目 | 分裂防止 |
| `main_ingredients` | 食材マスタの語彙のみ使用可(1〜4個)。調味料は含めない(`other_ingredients` へ) | REQUIREMENTS.mdのサンプルにあった醤油・みりん等は除外に変更。マッチング精度優先 |
| `method` フィールド | 追加。固定リスト: 焼く/炒める/煮る/蒸す/揚げる/和える・生食/汁物・鍋/ごはん・麺 | 同じ食材セットでも複数系統のレシピを出すため、調理法の偏りを管理 |
| `genre` | 固定リスト: 和食/洋食/中華/韓国・エスニック | 「スープ・鍋」「時短」はgenreではなくmethodとtime_minutesで表現 |
| データ形式 | JSONではなくJS定数ファイル(`data/ingredients.js`, `data/recipes.js`) | `file://` 直開きでfetchが失敗するのを避け、`<script src>` で完全オフライン動作を保証 |

## マッチング仕様の変更(開発中に更新)

| 項目 | 決定 | 理由・経緯 |
|---|---|---|
| 選択可能数 | 最小1個・最大5個(当初は最小3個) | 1食材からでも探せるようにしたいとのユーザー要望 |
| 2個以上選択時 | 選択食材のうち2つ以上を含むレシピを優先表示。ただし2つ一致が3件未満のときは1つ一致を補足表示するフォールバックあり | ①結果が空にならないUX ②「2つとも含む」体験の両立 |
| レシピ件数の方針 | 「件数より精度」を維持しつつ、②を実感できるよう日常の定番ペアを各3件以上に重点強化。目標500ではなくカバレッジ重視で約300件に着地 | ユーザーは500件案も検討したが、闇雲な水増しより定番ペア強化が費用対効果高いと合意 |
| 現状の規模 | 298件。全90食材が主要食材として3件以上に登場。定番27ペアはフォールバックなしで2つ一致が3件以上並ぶ | `tools/validate.js` で自動検証(3件未満食材・ペアカバレッジを計測) |

関連メモリ: [[fukusai-first-usage]](副菜メインで活用したい)

## ユーザーの好み・進め方

- 全ステップを最初に提示 → 1ステップずつ確認して進める
- レシピは件数が多いため、一括生成ではなくバッチ単位で品質確認しながら進める
